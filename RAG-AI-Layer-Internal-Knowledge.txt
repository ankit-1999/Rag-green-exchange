RAG AI Layer Internal Knowledge Document
Project: Green Marketplace RAG AI
Version: 1.0
Date: 2026-07-22

============================================================
1) Purpose Of This Document
============================================================
This file explains how the AI/RAG layer works internally in this application.
It is designed to be ingested into the same RAG index so the assistant can
answer meta questions such as:
- How does the RAG pipeline work?
- Which components are used?
- Which model names are used?
- How are demand and prediction questions answered?
- When does the system use live marketplace API data?


============================================================
2) High-Level Architecture
============================================================
The system combines two data channels:
1. Document RAG channel (S3 documents -> chunks -> embeddings -> OpenSearch)
2. Live marketplace channel (read-only public API tools)

Final answers are generated by Bedrock LLM using grounded context from one or
both channels.

Main orchestration service:
- app/services/rag_service.py

Primary support services:
- app/services/document_service.py      (ingestion write path)
- app/services/chunker_service.py       (text chunking)
- app/services/opensearch_service.py    (vector index/search)
- app/services/bedrock_service.py       (embeddings + generation + API planner)
- app/services/marketplace_api_service.py (live API execution)
- app/services/analytics_service.py     (deterministic analytics/prediction)
- app/services/prompt_service.py        (grounded answer prompt)
- app/services/tool_registry.py         (approved tool catalog)


============================================================
3) Models And Core Config
============================================================
Configured in app/config.py:
- Embedding model: amazon.titan-embed-text-v2:0
- LLM model: amazon.nova-micro-v1:0
- Embedding dimension: 1024
- LLM max output tokens default: 3200
- LLM temperature default: 0.0
- Planner max tokens default: 700
- Planner temperature default: 0.0

RAG retrieval and chunk settings:
- OPENSEARCH_TOP_K default: 8
- CHUNK_SIZE_TOKENS default: 400
- CHUNK_OVERLAP_TOKENS default: 100

Note on chunking strategy:
- Current chunker is character-window based with token estimate:
  1 token is approximated as about 4 characters.
- This is simple and fast; exact tokenizer chunking is not currently used.


============================================================
4) Document Ingestion Pipeline (Write Path)
============================================================
Entry endpoint:
- POST /documents/upload

Flow:
1. Validate request metadata (document name, type, S3 URI).
2. Ensure OpenSearch index exists.
3. Download file bytes from S3.
4. Extract text:
   - .txt and .md are decoded as UTF-8 text.
   - .pdf path exists, but real PDF parsing is not fully implemented in this
     codebase; it currently attempts decode fallback.
5. Split text into overlapping chunks.
6. Generate embedding vector per chunk using Titan embeddings.
7. Index each chunk + metadata + vector into OpenSearch.
8. Store document metadata in memory for listing/get operations.

Chunk identity behavior:
- Document ID is deterministic from S3 URI hash.
- Chunk IDs follow pattern: <document_id>_chunk_<index>
- Re-ingesting same S3 URI reuses IDs; search layer also deduplicates similar
  duplicate hits from prior ingestions.


============================================================
5) Query Pipeline (Read Path)
============================================================
Entry endpoint:
- POST /query
Payload:
- question (required)
- top_k (optional, 1..20)

Core orchestration in rag_service.answer_question:

Step A: Select route style
- If question looks like architecture/internal design documentation intent
  (including words like lld, hld, low level design, high level design,
  workflow, architecture, smart contract module, requirements doc), the system
  uses retrieval-only document mode first.

Step B: For general marketplace/data questions
- Use Bedrock planner to infer intent and required live tools.
- Execute validated read-only tool calls.
- Run deterministic analytics/prediction/recommendation calculations.

Step C: Retrieve RAG chunks
- Embed user question.
- kNN search in OpenSearch with top_k.
- Build source objects for citation metadata.

Step D: Build grounded prompt
- Prompt receives:
  - RAG context chunks
  - compact API context (if used)
  - analytics/prediction/recommendation outputs
- Prompt enforces strict grounding and anti-hallucination rules.

Step E: Generate final answer
- Use Bedrock Nova Micro to produce safe HTML fragment.
- If generation fails, fallback text is returned.


============================================================
6) When Live API Data Is Used
============================================================
Live data is used when planner intent is not "none" and the question asks for
marketplace state, metrics, trends, predictions, or recommendations.

Read-only tools available:
- get_all_listings
- get_active_listings
- get_all_purchases

Important safeguards:
- Only approved tools can run.
- Only public GET APIs are allowed.
- Unsupported filters are dropped.
- Multi-page fetch is supported for full coverage.
- If live data is required but APIs return no usable records/aggregates,
  response is marked insufficient and the system does not present static
  document examples as live facts.

Planner intents (examples):
- current_supply
- supply_mix
- historical_supply
- historical_demand
- demand_and_supply
- average_selling_price
- demand_supply_ratio
- market_balance
- supply_stability
- price_volatility
- supply_by_location
- marketplace_summary
- demand_prediction
- price_prediction
- shortage_prediction
- seller_recommendation
- buyer_recommendation


============================================================
7) How Demand Questions Are Answered
============================================================
For demand-related intents, the system uses deterministic calculations in
analytics_service, not LLM arithmetic.

Typical demand-related logic:
- historical_demand:
  completed purchase records are grouped by source; demand totals and shares
  are computed.

- demand_and_supply:
  combines listing and purchase datasets to derive:
  total supply, remaining supply, sold supply, realized demand.

- demand_supply_ratio:
  computes completed demand kWh divided by listed supply kWh.

The LLM role is explanation and presentation only. It should not invent or
recalculate core numbers that analytics already produced.


============================================================
8) How Prediction And Forecast Questions Are Answered
============================================================
Prediction questions are computed in analytics_service, then explained by LLM.

A) Demand prediction
- Uses recursive four-period weighted moving average style forecasting.
- Weight pattern referenced in rendering logic: 0.1, 0.2, 0.3, 0.4.
- Works from historical completed demand series.

B) Price prediction
- Uses backend monthly trend data when available.
- Falls back to weekly/raw-record derived approach when needed.

C) Shortage prediction
- Combines forecast demand, forecast new supply, and current active supply.
- Conceptual gap formula used by renderer:
  projected gap = forecast new supply + current active supply - forecast demand
- Shortage flag is driven by whether demand exceeds available projected supply.

Confidence and limitations:
- Confidence labels are produced (high/medium/low/insufficient_data).
- Limitations are attached when records are sparse/partial/failing.


============================================================
9) Recommendation Question Handling
============================================================
Two recommendation types exist:
- seller_recommendation
- buyer_recommendation

Deterministic scoring approach:
- Seller opportunity scoring uses weighted factors including demand growth,
  demand-to-supply ratio, price strength, and lower active saturation.
- Buyer listing scoring uses factors including price, demand, quantity,
  and recency.

LLM explains why, but does not invent missing market values.


============================================================
10) Prompt Grounding Rules (Critical)
============================================================
Prompt layer explicitly enforces:
- Use only API context and RAG context.
- Do not invent values, dates, sources, locations, or calculations.
- Prefer deterministic analytics outputs where provided.
- If static RAG examples conflict with live API derived values, live values win.
- If no live data exists for a live-data question, state that limitation.

Response shape:
- Returned answer is sanitized HTML fragment, not markdown.
- Source details are returned separately in QueryResponse.sources.


============================================================
11) Query Answer Modes
============================================================
Possible answer_mode values:
- retrieval_only
- retrieval_plus_api
- api_only
- insufficient_data

Meaning:
- retrieval_only: answer from RAG chunks only.
- retrieval_plus_api: combined docs + live marketplace context.
- api_only: live API and analytics context without chunk retrieval.
- insufficient_data: not enough grounded data to answer reliably.


============================================================
12) Why This Design Is Reliable For LLD/HLD/Workflow Docs
============================================================
- .md documents are ingested as UTF-8 text and chunked like normal text.
- Chunk overlap keeps section continuity for long technical docs.
- Architecture-question detector routes LLD/HLD/workflow questions toward
  document retrieval mode.
- Live-data guard prevents mixing stale examples with current market claims.


============================================================
13) Known Limitations
============================================================
1. Chunking is approximate-token based (chars/4), not model-token exact.
2. PDF extraction path is limited compared with true PDF parsers.
3. Very large chunks can reduce retrieval precision for highly structured docs.
4. If documents are re-ingested with changed content but same S3 URI, old index
   conditions may require cleanup depending on ingestion pattern.


============================================================
14) Recommended Tuning For Detailed Technical QA
============================================================
Suggested defaults for architecture-heavy docs:
- CHUNK_SIZE_TOKENS: 350 to 500 (current default 400)
- CHUNK_OVERLAP_TOKENS: 80 to 120 (current default 100)
- top_k for deep explanation: 8 to 12
- LLM temperature: 0.0 for factual consistency

Rule of thumb:
- Increasing chunk size too much can hurt precision.
- For better detail, usually increase top_k before aggressively increasing
  chunk size.


============================================================
15) Quick FAQ For Meta Questions
============================================================
Q: Which LLM is used for final answer generation?
A: amazon.nova-micro-v1:0

Q: Which model is used for embeddings?
A: amazon.titan-embed-text-v2:0

Q: Does AI do direct marketplace writes?
A: No. AI layer uses read-only marketplace APIs only.

Q: Are predictions pure LLM guesses?
A: No. Forecast calculations are deterministic in analytics_service; LLM mainly
   explains results.

Q: When will it refuse to provide live market facts?
A: When planner requires live data but APIs return no usable records/aggregates,
   response is insufficient_data.

Q: Can it answer "Explain the LLD" from documents?
A: Yes, if LLD file is ingested and retrievable from OpenSearch.
